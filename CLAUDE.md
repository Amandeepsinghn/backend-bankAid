# Backend — Claude Code Instructions

## Stack
Express + TypeScript (strict) + Drizzle ORM + PostgreSQL (Supabase via DATABASE_URL) + Razorpay + ZeptoMail (email OTP)

```bash
npm run dev              # ts-node-dev on port 4000
npm run test             # Vitest

npx drizzle-kit generate # Generate migration files
npx drizzle-kit migrate  # Apply migrations
```

## Environment variables
Only these are needed in `.env`:
- `JWT_SECRET` — single secret for all token signing (auth + reset, distinguished by `type` field in payload)
- `DATABASE_URL` — Supabase Postgres connection string
- `ZEPTOMAIL_API_KEY`, `ZEPTOMAIL_FROM_EMAIL`, `ZEPTOMAIL_FROM_NAME` (optional, defaults to "Unfreeze"), `ZEPTOMAIL_API_URL` (optional, defaults to the India region endpoint) — registration and password-reset OTP emails. Required for the app to function at all — registration itself depends on email delivery
- `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`

## Database schema (5 tables)
```
profiles ──< subscriptions ──< form_submissions ──< letter_status
profiles ──< form_submissions (direct FK too)
otps (standalone — OTP codes with expiry)
```

## Request lifecycle (never skip a layer)
```
route.ts → requestHygiene.ts (Zod) → verifyToken → requireSubscription → routeController.ts → service.ts → db.ts
```

## Generating a new feature — always produce all 7 files together
| File | Responsibility |
|---|---|
| `<feature>.route.ts` | Express Router, apply middleware, wire handlers, Swagger JSDoc annotations |
| `<feature>.routeController.ts` | Thin — call service, return response |
| `<feature>.service.ts` | All business logic |
| `<feature>.requestHygiene.ts` | Zod schemas for every request body/param/query |
| `<feature>.types.ts` | TypeScript types & interfaces for the feature |
| `<feature>.db.ts` | All Drizzle calls — nowhere else |
| `<feature>.test.ts` | Vitest unit + integration tests |

Register the router in `src/app.ts` after generating files.

## Hard rules
- `any` is never acceptable — strict TypeScript everywhere
- Raw `req.body` never goes to Drizzle — always pass through Zod first
- DB calls only in `*.db.ts` files
- Logic only in `*.service.ts` files
- Form routes → `requireSubscription` middleware
- Case limits (escalation iterations): basic=1, standard=2, premium=3 — enforced server-side in form.service.ts
- Never commit secrets — use `.env` (gitignored), update `.env.example`

## Error handling
Use `asyncHandler` wrapper from `lib/asyncRouter.ts` — never bare try/catch in controllers. Throw `AppError` with HTTP status from service layer. Global `errorHandler` middleware catches everything. Zod errors are caught in app.ts and return 400 with field-level details.

## Payments (Razorpay)
1. Backend creates order → returns `order_id` to client
2. Client completes payment → sends `payment_id` + `signature` to backend
3. Backend verifies signature via `lib/razorpay.ts` — never trust client confirmation

## Tier validation
- 3 tiers: basic (below ₹10K → ₹199, 1 iteration), standard (₹10K–₹1L → ₹299, 2 iterations, Value Pack — Telegram support), premium (above ₹1L → ₹499, 3 iterations, call + Telegram support)
- Validation runs server-side in form.service.ts before persisting submission
- If stuck amount exceeds paid tier range → BLOCK with upgrade message
- If stuck amount is below paid tier range (overpaid) → allow silently

## Auth flow
- Two token types, both signed with `JWT_SECRET`:
  - **Access token** — JWT, `type: 'auth'`, **15-minute** expiry, sent as `Authorization: Bearer` header
  - **Reset token** — JWT, `type: 'reset'`, **15-minute** expiry, single-use for password reset only
- **Refresh token** — opaque 96-char hex (`crypto.randomBytes(48).toString('hex')`), **30-day** expiry, stored as SHA-256 hash in `refresh_tokens` table; rotated on every use
- Client stores tokens in `expo-secure-store` (iOS Keychain / Android Keystore) — the mobile equivalent of httpOnly cookies; do NOT return to web clients without switching to httpOnly cookies
- There is no phone OTP / SMS verification anywhere in this app — email OTP (via ZeptoMail) is the only verification channel, full stop. Don't reintroduce Twilio or a phone-verification flow without being asked
- Register: email + password + fullName → OTP emailed via ZeptoMail (`email_verification` type) → `/verify-email-otp` verifies it, marks `profiles.emailVerified`, issues access + refresh tokens. `/resend-email-otp` (email only) reissues it while unverified
- Login: email + password → access + refresh tokens issued, blocked with 403 if `profiles.emailVerified` is false
- Forgot password: email + phone → if the account has a phone on file it must match (most won't, since phone isn't collected at registration), then an OTP is emailed to that address via ZeptoMail → verify → reset token → new password. Calling `/forgot-password` again acts as its own resend — there's no separate resend endpoint for this flow
- Every OTP send (register, resend-email-otp, forgot-password) goes through `issueOtp()` in auth.service.ts: enforces a 60-second resend cooldown per identifier+type, and immediately expires any previously issued unverified code for that identifier+type so only the newest one works. Natural expiry is still 10 minutes
- OTPs stored in `otps` table with 10-minute expiry, marked verified after use (the `phone` column doubles as a generic identifier — always an email address in practice now, named `phone` only to avoid an ambiguous rename migration)
